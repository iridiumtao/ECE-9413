# ECE-9413 Assignment 2: SumCheck Prover in JAX

## What This Is

An implementation of the prover-side logic for the SumCheck protocol using JAX.
The prover takes a multilinear polynomial expressed as an evaluation table, runs
`num_rounds` rounds of the protocol (each folding one Boolean variable with a random
challenge), and returns the initial claim together with the per-round univariate
evaluations.

## Core Value

Correctly produce `(claim0, round_evals)` that passes the harness for all required
32-bit base expressions (`a`, `a*b`, `a*b+c`, `a*b*c`) on vars4, vars16, and vars20.

## Context

- **Harness**: `student.py` is the only file students edit; the harness in `provided.py`,
  `sumcheck_utils.py`, and `tests/` is read-only.
- **Expression format**: `list[list[str]]` — outer list is additive terms, inner is
  multiplicative factors. E.g., `a*b + c` → `[["a", "b"], ["c"]]`.
- **eval_tables**: dict from variable name → JAX array of MLE evaluations.
- **Challenges**: `num_rounds - 1` values passed to `sumcheck_32` (final verifier-only
  challenge excluded).
- **Rounds must be serialized**: cannot parallelise across rounds.
- **Prime**: `MAX_PRIME_32 = 4294967291` fits in uint32, but intermediate products
  can overflow — must use uint64 for multiplication then reduce.
- **Debug tool**: `provided.expression_round_trace()` available for vars4 only.
- **JAX x64**: `jax_enable_x64 = True` is set globally; use `jnp.uint64` or
  `jnp.int64` for intermediates.
- **Platform**: default CPU; GPU/TPU selectable via `JAX_PLATFORMS`.

## Constraints

- **API**: Frozen dispatchers (`sumcheck`, `mod_add`, …) must not be renamed or changed.
- **File**: Only `student.py` is edited.
- **Timeline**: Due within 1 week.
- **Round serialization**: Each round depends on the previous challenge — cannot be
  parallelized at the round level (but intra-round vectorization with `vmap`/`jit` is fine).
- **Performance target**: JAX JIT + vmap — design for GPU/TPU from the start.
